# AGENTS.md — homelab-iac

## TL;DR

Infrastructure-as-Code repo managing a homelab on a **single-node Proxmox VE** hypervisor (`192.168.40.10`). Terraform provisions Rocky Linux 10 VMs from a cloud-init template; Ansible configures them into Docker hosts running self-hosted services (Pi-hole, Traefik, Portainer, Uptime Kuma, Homepage, Prometheus stack, Paperless-ngx, Valheim). The `Makefile` is the unified CLI — every operation has a `make` target.

## Key conventions

### Terraform conventions

- **VM module is reusable**: `terraform/modules/vm/` — accepts a map of VMs, creates `proxmox_virtual_environment_vm` resources with cloud-init.
- **Declarative VM definitions**: All VMs live in `locals.tf` as a flat HCL map (`local.vms`). To add a VM, add a new entry to this map — do NOT modify the module or main.tf.
- **IP auto-generated**: `"${local.network_prefix}.${each.value.vm_id}${local.network_cidr}"` — no manual IP entry needed.
- **Tags drive Ansible groups**: Proxmox tags (excluding `terraform`) become Ansible inventory groups. Underscores in tags become hyphens in group names.
- **Full clone from template**: `clone { vm_id = var.template_id, full = true }` with `lifecycle { ignore_changes = [clone] }` so Terraform doesn't try to re-clone.
- **OS type**: Always `l26` (Linux 2.6+) for all VMs.
- **Provider lock**: `.terraform.lock.hcl` is committed (pin `bpg/proxmox` to exactly `v0.104.0`).

### Ansible conventions

- **Everything is a role**: Playbooks are thin wrappers that compose roles. No inline tasks in playbooks.
- **Common role pattern**: `lvm-resize` → `docker` → `{service-role}` (with optional `server-hardening` for internet-facing VMs).
- **Fact caching**: Enabled (JSON files in `.ansible_facts_cache/`, TTL 2 hours). `gathering = smart`.
- **Serial execution**: `serial: "50%"` on update playbook, with `any_errors_fatal: false`.
- **Intelligent reboot**: `reboot_policy: if-needed` — uses `needs-restarting` or checks `/var/run/reboot-required`.
- **Docker Compose v2**: Always use the `docker compose` plugin, never `docker-compose` (legacy v1).
- **Handlers**: Use `docker compose restart` for image changes, `docker compose down && docker compose up -d` for config file changes.
- **Service directories**: All compose files deployed to `/opt/{service}/`.
- **Firewalld**: Services that need network access open ports via firewalld tasks (not in the docker role, but alongside the service role).
- **Inventory generation**: `make inventory` runs `terraform output -raw ansible_inventory` and writes to `hosts.generated`. Do NOT manually edit `hosts.generated`.

### Docker conventions

- **Traefik network**: Named bridge network `proxy` — Traefik and all services that need reverse proxying connect to it.
- **Image tags**: `:latest` for most services, pinned versions for Traefik (`v3.1`), Portainer (`2.21.5`), and monitoring stack components.
- **Data persistence**: Data directories on host filesystem, either on the OS disk or mounted at `/srv/data` for data-disk VMs.
- **Secrets in containers**: Passed via environment variables sourced from Ansible vault, or auto-generated on first run (Paperless).
- **Exposed ports**: Traefik HTTP on host port 80 (HTTPS/443 is commented out — no TLS, internal-only lab).

## Adding a new VM (step-by-step)

1. **Edit** `terraform/environments/prod/locals.tf` — add a new entry to the `local.vms` map:

    ```hcl
    {name}-01 = {
      vm_id     = {next-available-id}
      cpu_cores = 4
      memory_mb = 8192
      disks = [
        { datastore_id = "local-lvm", size = 50, interface = "scsi0" },
        # Optional data disk:
        # { datastore_id = "datav1", size = 200, interface = "scsi1" },
      ]
      tags = ["terraform", "{tag1}", "{tag2}"]
    }
    ```

    - VM names MUST follow `{purpose}-{sequence}` pattern (e.g., `media-01`).
    - VMID MUST fit the numbering scheme and be ≤ 219 for clean IP addressing.
    - Always include `terraform` in tags (it's filtered out of Ansible groups).
    - Additional tags become Ansible host groups (e.g., `["media", "jellyfin"]` → groups `[media]`, `[jellyfin]` in inventory).

2. Run Terraform:
    ```
    make plan    # Review what will be created
    make apply   # Provision the VM
    ```

3. Regenerate the Ansible inventory:
    ```
    make inventory
    ```

4. Create an Ansible playbook and roles for the new service, following the existing patterns.

5. Deploy the service:
    ```
    make {service-name}
    ```

## Adding DNS + Traefik routes for a new service

When a new service needs a `*.lab` domain accessible through Traefik, three files must be updated.

All `*.lab` domains resolve to Traefik's IP (`192.168.40.142`, proxy-01). Traefik then reverse-proxies to the actual service IP and port.

### Step 1: Pi-hole DNS — `FTLCONF_dns_hosts` (PRIMARY, env var in docker-compose)

**File**: `ansible/roles/pihole/templates/docker-compose.yml.j2`

Add the new domain to `FTLCONF_dns_hosts`:

```yaml
FTLCONF_dns_hosts: |-
  192.168.40.141 pihole.lab
  192.168.40.142 traefik.lab
  192.168.40.142 kuma.lab
  ...
  192.168.40.142 {service}.lab          # ← add here, pointing to proxy-01 (.142)
```

This is the **primary** DNS mechanism — Pi-hole's FTL reads this env var and serves the records directly. Without this entry, DNS resolution will fail.

### Step 2: Pi-hole DNS — `custom.list.j2` (secondary, dnsmasq custom)

**File**: `ansible/roles/pihole/templates/custom.list.j2`

Add the same entry for completeness:

```
192.168.40.142 {service}.lab
```

This file is mounted as a dnsmasq custom config. Keep it in sync with `FTLCONF_dns_hosts`.

### Step 3: Traefik dynamic config

**File**: `ansible/roles/traefik/templates/dynamic.yml.j2`

Add a **router** and a **service** block:

```yaml
# In the routers: section:
    {service}:
      rule: Host(`{service}.lab`)
      service: {service}
      entryPoints:
        - web

# In the services: section:
    {service}:
      loadBalancer:
        servers:
          - url: "http://{{ hostvars['{target-host}']['ansible_host'] | default('192.168.40.{last-octet}') }}:{port}"
```

Use the correct `target-host` (Ansible inventory hostname) and `last-octet`/port matching the service. See existing examples:
- `app-01` / `.143` for services on the apps VM
- `monitor-01` / `.145` for monitoring services
- `dns-01` / `.141` for DNS-related services

### Step 4: Deploy

```bash
make dns    # Pi-hole picks up new DNS entries (container is recreated, not just restarted)
make proxy  # Traefik picks up new routes (dynamic config has watch: true, but handler does down+up)
```

### Verification

```bash
# DNS resolution (must query Pi-hole directly unless your machine uses it)
dig +short {service}.lab @192.168.40.141

# Traefik routing
curl -s -o /dev/null -w "%{http_code}" -H "Host: {service}.lab" http://192.168.40.142/

# List all Traefik routers
curl -s http://192.168.40.142:8080/api/http/routers | python3 -c "import sys,json; [print(r['rule'],'→',r['service']) for r in json.load(sys.stdin)]"
```

## Terraform ↔ Ansible bridge explained

The glue between Terraform and Ansible is `terraform/environments/prod/outputs.tf`:

1. Terraform manages VMs with Proxmox **tags** (e.g., `["terraform", "dns", "network"]`).
2. The `ansible_inventory` output iterates over all VM tags, creating INI-format group sections.
3. The `terraform` tag is **explicitly filtered out** — it would be useless as an Ansible group.
4. Tag names with hyphens get underscores in group names: tag `server-hardening` → group `[server_hardening]`.
5. `make inventory` runs `terraform output -raw ansible_inventory > hosts.generated`.
6. Ansible playbooks target these auto-generated groups (e.g., `hosts: dns`, `hosts: proxy`).

Example generated inventory:

```ini
[all:vars]
ansible_user=homelab
ansible_python_interpreter=/usr/bin/python3

[all]
dns-01 ansible_host=192.168.40.141
proxy-01 ansible_host=192.168.40.142

[network]
dns-01 ansible_host=192.168.40.141
proxy-01 ansible_host=192.168.40.142

[dns]
dns-01 ansible_host=192.168.40.141

[proxy]
proxy-01 ansible_host=192.168.40.142
```

## Key architectural decisions

1. **Proxmox as the hypervisor** — Single node, no clustering. Uses `bpg/proxmox` provider which has native cloud-init, multi-disk, and clone support.

2. **Rocky Linux 10** — RHEL-compatible, cloud-init native. All VMs derive from VMID 9000 template.

3. **Flat network, no VLANs** — `/24` subnet with IP=VMID convention. Simple, mnemonic. No network segmentation.

4. **Traefik with file-based dynamic config** — Static list of services in a Jinja2 template. `exposedByDefault: false` prevents accidental exposure. No TLS (internal-only lab, HTTPS entrypoint is commented out).

5. **Docker Compose everywhere** — No Kubernetes. All services are Docker compose stacks. No container orchestration beyond basic `docker compose`.

6. **No CI/CD** — All operations are manual via `make` targets. No GitHub Actions, no pipelines, no pre-commit hooks (except git's sample hooks).

7. **Local Terraform state** — No remote backend (no S3/Azure/GCS). State file is local at `terraform.tfstate` (gitignored).

8. **Monitoring with static targets** — Prometheus scrape configs use hardcoded IPs, not service discovery.

9. **Server hardening for internet-facing VMs** — `server-hardening` role (fail2ban, dnf-automatic, firewalld, SSH hardening, sysctl) applied ONLY to `games-01` and `monitor-01`.

10. **Polish localization** — Dashboard descriptions, timezone (Europe/Warsaw). Obviously the operator is Polish; notes and descriptions may be in Polish.

## Bootstrapping a new Proxmox host

1. SSH to Proxmox: `ssh root@192.168.40.10`
2. Run `scripts/proxmox-create-rocky-template.sh` — downloads Rocky Linux 10 qcow2, creates VMID 9000 template with cloud-init.
3. Run `make datastore` — creates the `datav1` storage backend.
4. Follow the quickstart in `readme.md`.
